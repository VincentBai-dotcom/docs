---
title: "Commands: Defining Player Actions with Validation"
description: "Commands are typed player actions with an input schema, optional availability guard, a validate function, and an execute function that mutates game state."
---

A command represents a single player action — drawing a card, placing a piece, spending resources. Each command definition bundles together its identifier, the shape of its input, validation logic that runs before execution, and an execute function that applies the action to game state. You build commands with `createCommandFactory()` and attach them to stages.

## Creating a command factory

Call `createCommandFactory()` parameterised with your game's root state class. This returns a builder function you call to start defining each command:

```typescript
import { createCommandFactory } from "tabletop-engine";
import type { MyGameState } from "./state.ts";

const command = createCommandFactory<MyGameState>();
```

## Defining a command

Pass a `commandId` and a `commandSchema` (built with `t.object`) to the factory, then chain `.validate()`, `.execute()`, and optionally `.isAvailable()`:

```typescript
import { t } from "tabletop-engine";

const drawCardCommand = command({
  commandId: "draw_card",
  commandSchema: t.object({
    deckId: t.string(),
  }),
})
  .isAvailable(({ game, actorId }) => {
    return game.getPlayer(actorId).hand.length < 7;
  })
  .validate(({ game, command }) => {
    const { deckId } = command.input;
    if (!game.hasCards(deckId)) {
      return { ok: false, reason: "deck_empty" };
    }
    return { ok: true };
  })
  .execute(({ game, command, rng, emitEvent }) => {
    const { actorId, input } = command;
    const card = game.drawCard(input.deckId, rng);
    game.getPlayer(actorId).hand.push(card.id);
    emitEvent({ category: "domain", type: "card_drawn", payload: { actorId, cardId: card.id } });
  })
  .build();
```

## Command lifecycle

Each command goes through the following steps when the executor receives it:

<Steps>
  <Step title="isAvailable check">
    If you defined `.isAvailable()`, the executor calls it first. A `false` return causes immediate rejection without running validation. Use this for quick structural checks — for example, whether the command type is valid for the current active player. This step is optional.
  </Step>
  <Step title="validate">
    The executor calls `.validate()` with `{ game, runtime, command }`. Return `{ ok: true }` to allow the command, or `{ ok: false, reason: "..." }` to reject it with a machine-readable reason string. Validation runs against a read-only view of game state.
  </Step>
  <Step title="execute">
    Once validation passes, the executor calls `.execute()` inside a transaction. You receive a mutable `game` facade instance and can call methods, mutate fields, and emit events. If you throw an error inside `execute`, the transaction is rolled back.
  </Step>
</Steps>

## Command properties

<ParamField path="commandId" type="string" required>
  A unique string identifier for this command across the entire game. The executor uses this to look up the command definition when a player submits an action. All command IDs must be unique — `GameDefinitionBuilder.build()` throws `duplicate_command_id` if two different command objects share an ID.
</ParamField>

<ParamField path="commandSchema" type="ObjectFieldType" required>
  The schema for the command's `input` field, built with `t.object()`. The static type from the schema is inferred and flows into the `validate` and `execute` contexts so `command.input` is fully typed.
</ParamField>

<ParamField path="isAvailable" type="(context) => boolean">
  Optional availability guard. Receives `{ game, runtime, commandType, actorId }`. Return `false` to prevent this command from being submitted by the current actor in the current state. This is evaluated before validation.
</ParamField>

<ParamField path="validate" type="(context) => ValidationOutcome" required>
  Receives `{ game, runtime, command }`. Return `{ ok: true }` or `{ ok: false, reason: string }`. Game state is read-only. Do not mutate state here.
</ParamField>

<ParamField path="execute" type="(context) => void" required>
  Receives `{ game, runtime, command, rng, emitEvent }`. Mutate the `game` facade directly. Call `emitEvent()` to emit structured domain events. Throw an error to abort and roll back.
</ParamField>

## Attaching commands to stages

Commands are attached to stages, not to the game definition directly. The `GameDefinitionBuilder` collects every command reachable from the stage graph automatically:

```typescript
const turnStage = stage("player-turn")
  .singleActivePlayer()
  .activePlayer(({ game }) => game.currentPlayerId)
  .commands([drawCardCommand, playCardCommand, passCommand])
  .nextStages(() => ({ next: endTurnStage }))
  .transition(({ nextStages }) => nextStages.next)
  .build();
```

## Command discovery

Some commands have input that emerges from multi-step selection flows — for example, choosing a card from the board requires first picking a level, then picking a specific card. Command discovery models this as a series of steps before committing a full command.

Mark a command as discoverable with `.discoverable()` and define the steps using a step factory:

```typescript
const buyFaceUpCardCommand = command({
  commandId: "buy_face_up_card",
  commandSchema: t.object({
    level: t.number(),
    cardId: t.number(),
  }),
})
  .discoverable((step) => [
    step("select_card")
      .initial()
      .input(t.object({ selectedCardId: t.optional(t.number()) }))
      .output(t.object({ level: t.number(), cardId: t.number() }))
      .resolve(({ actorId, game, discovery }) => {
        const draft = discovery.input;

        if (draft.selectedCardId !== undefined) {
          const card = game.getCard(draft.selectedCardId);
          return { complete: true, input: { level: card.level, cardId: card.id } };
        }

        // Return options for the client to present
        return game.board.faceUpCards
          .filter((card) => game.getPlayer(actorId).canAfford(card))
          .map((card) => ({
            id: String(card.id),
            output: { level: card.level, cardId: card.id },
            nextInput: { selectedCardId: card.id },
            nextStep: "select_card",
          }));
      })
      .build(),
  ])
  .validate(({ game, command }) => {
    // validation logic
    return { ok: true };
  })
  .execute(({ game, command }) => {
    // execution logic
  })
  .build();
```

Each step defines:
- `stepId` — a string key for this step.
- `.initial()` — marks this as the first step.
- `.input(schema)` — the schema for the partial input accumulated so far.
- `.output(schema)` — the schema for the data the client receives per option.
- `.resolve(fn)` — returns either an array of options for the client to select from, or `{ complete: true, input: {...} }` when the full command input is known.

<Info>
  Discovery is a client-facing feature. The executor exposes a `discover()` method that the client calls to walk through the steps. When the step resolves with `complete: true`, the client submits the full command.
</Info>

## Real example: taking three gems in Splendor

```typescript
import { createCommandFactory, t } from "tabletop-engine";
import type { SplendorGameState } from "./state.ts";

const command = createCommandFactory<SplendorGameState>();

const takeThreeDistinctGemsCommand = command({
  commandId: "take_three_distinct_gems",
  commandSchema: t.object({
    colors: t.array(t.string()),
  }),
})
  .isAvailable(({ game }) => {
    const gemColors = Object.entries(game.bank).filter(
      ([color, count]) => color !== "gold" && count > 0,
    );
    return gemColors.length >= 3;
  })
  .validate(({ game, command }) => {
    const { colors } = command.input;

    if (colors.length !== 3) {
      return { ok: false, reason: "three_colors_required" };
    }

    if (new Set(colors).size !== 3) {
      return { ok: false, reason: "colors_must_be_distinct" };
    }

    for (const color of colors) {
      if (game.bank[color] <= 0) {
        return { ok: false, reason: "token_color_unavailable" };
      }
    }

    return { ok: true };
  })
  .execute(({ game, command, emitEvent }) => {
    const { actorId, input } = command;
    const player = game.getPlayer(actorId);

    for (const color of input.colors) {
      game.bank.adjustColor(color, -1);
      player.tokens.adjustColor(color, 1);
    }

    emitEvent({
      category: "domain",
      type: "gems_taken",
      payload: { actorId, colors: input.colors },
    });
  })
  .build();
```
