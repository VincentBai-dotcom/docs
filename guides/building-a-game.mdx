This guide walks you through building a complete game with tabletop-engine from scratch. You'll model game state, define commands with validation, wire up stage progression, and produce an executor you can drive from a server or test harness. The Splendor reference implementation lives in `examples/splendor/engine/` and is used throughout as a concrete illustration.

<Steps>

<Step title="Plan your game state">

Start by deciding what data your game needs to track. Draw a clear boundary between data that belongs to all players (the board, the bank, turn order) and data that belongs to individual players (their hand, tokens, score).

For a card game like Splendor you might identify:

- **Game-level state** — player order, shared bank, the board of face-up cards, end-game tracking, and winner resolution
- **Player state** — the player's ID, their tokens, reserved cards, purchased cards, and claimed nobles
- **Board state** — face-up card slots by level, shuffled decks by level, and available nobles

Separating these into individual state classes keeps logic focused and makes visibility rules much easier to express later.

</Step>

<Step title="Define state facades with @State() and @field()">

Each state class is a plain TypeScript class decorated with `@State()`. Every persisted property is annotated with `@field()` and a schema expression from `t`.

```typescript
import { field, State, t } from "tabletop-engine";

@State()
export class PlayerState {
  @field(t.string())
  id = "";

  @field(t.state(() => TokenCountsState))
  tokens!: TokenCountsState;

  @field(t.array(t.number()))
  reservedCardIds: number[] = [];

  @field(t.array(t.number()))
  purchasedCardIds: number[] = [];
}
```

Nested state classes use `t.state(() => NestedClass)` — the lambda avoids circular-reference issues at module load time. Use `t.record(t.string(), t.state(() => PlayerState))` to model a keyed map of player states.

The root state class should own the full game tree:

```typescript
@State()
export class GameState {
  @field(t.array(t.string()))
  playerOrder: string[] = [];

  @field(
    t.record(t.string(), t.state(() => PlayerState)),
  )
  players: Record<string, PlayerState> = {};

  @field(t.state(() => BoardState))
  board!: BoardState;
}
```

<Tip>
Add domain helper methods directly to your state classes — `getPlayer(id)`, `getNextPlayerId(current)`, `getScore()`. The executor calls your command logic against a hydrated instance of these classes, so methods are available in full.
</Tip>

</Step>

<Step title="Design your stage progression">

Stages define who can act and which commands are available at any point in the game. There are two main kinds:

- **`singleActivePlayer`** — one player acts. The stage declares an `activePlayer()` resolver that derives the current actor from game state and runtime metadata.
- **`automatic`** — no player input is needed. The stage runs a side-effecting `run()` block and transitions immediately.

Use `createStageFactory()` to get a typed `defineStage` helper scoped to your root state class, then wire stages together through `nextStages()` / `transition()`:

```typescript
import { createStageFactory } from "tabletop-engine";
import type { GameState } from "./state.ts";

const defineStage = createStageFactory<GameState>();

const checkVictoryStage = defineStage("checkVictory")
  .automatic()
  .run(({ game, runtime, emitEvent }) => {
    // resolve end-of-turn side effects
    game.resolveTurnEnd(getLastActingPlayerId(runtime), emitEvent);
  })
  .nextStages(() => ({
    gameEndStage,
    playerTurnStage,
  }))
  .transition(({ game, nextStages }) =>
    game.winnerIds ? nextStages.gameEndStage : nextStages.playerTurnStage,
  )
  .build();
```

For the `playerTurn` stage, the active player resolver inspects `runtime.progression.lastActingStage` to derive whose turn it is:

```typescript
const playerTurnStage = defineStage("playerTurn")
  .singleActivePlayer()
  .activePlayer(({ game, runtime }) => {
    const previousActorId = runtime.progression.lastActingStage
      ? getLastActingPlayerId(runtime)
      : null;
    return previousActorId
      ? game.getNextPlayerId(previousActorId)
      : game.playerOrder[0]!;
  })
  .commands(commands)
  .nextStages(() => ({
    checkVictoryStage,
    returnExcessiveTokensStage,
  }))
  .transition(({ game, command, nextStages }) => {
    if (game.getPlayer(command.actorId).getRequiredReturnCount() > 0) {
      return nextStages.returnExcessiveTokensStage;
    }
    return nextStages.checkVictoryStage;
  })
  .build();
```

<Note>
Stages reference each other through lazy getters (`() => stage`) to handle forward references. Build all stage objects before passing any of them to `GameDefinitionBuilder`.
</Note>

</Step>

<Step title="Write commands with validation and execution">

Each command is built with a typed command factory. The factory is scoped to your root state class, which gives you full type safety throughout `.validate()` and `.execute()`.

```typescript
import { createCommandFactory, t } from "tabletop-engine";
import type { GameState } from "./state.ts";

const defineCommand = createCommandFactory<GameState>();

const takeTwoSameGemsCommand = defineCommand({
  commandId: "take_two_same_gems",
  commandSchema: t.object({ color: t.string() }),
})
  .isAvailable(({ game, actorId }) => {
    // Return true only when the command can meaningfully be offered
    const player = game.getPlayer(actorId);
    return Object.entries(game.bank).some(
      ([color, count]) => color !== "gold" && count >= 4,
    );
  })
  .validate(({ game, command }) => {
    const { color } = command.input;

    if (!isGemColor(color)) {
      return { ok: false, reason: "invalid_color" };
    }

    if (game.bank[color] < 4) {
      return { ok: false, reason: "not_enough_tokens_for_double_take" };
    }

    return { ok: true };
  })
  .execute(({ game, command, emitEvent }) => {
    const { actorId, input: { color } } = command;
    const player = game.getPlayer(actorId);

    game.bank[color] -= 2;
    player.tokens[color] += 2;

    emitEvent({
      category: "domain",
      type: "gems_taken",
      payload: { actorId, color, count: 2 },
    });
  })
  .build();
```

The executor calls `.validate()` before `.execute()`. If validation returns `{ ok: false, reason }`, the command is rejected and state is not mutated. `.isAvailable()` controls whether a command appears in `listAvailableCommands()` at all — use it to filter commands that are structurally impossible given the current state.

<Warning>
The execute block runs against a working copy of state. Throw from `.execute()` only for programmer errors (unexpected state that should never occur), not for expected validation failures — those belong in `.validate()`.
</Warning>

</Step>

<Step title="Configure setup">

The setup function receives a fresh instance of your root state class, a seeded `RNGApi`, and whatever typed input you declare with `.setupInput()`. Use it to perform all initial randomization and population:

```typescript
import type { RNGApi } from "tabletop-engine";
import type { GameState } from "./state.ts";

export function setupGame(
  game: GameState,
  rng: RNGApi,
  playerIds: readonly string[],
): void {
  game.initializePlayers(playerIds);
  game.initializeBank(playerIds.length);

  for (const level of [1, 2, 3] as const) {
    const shuffled = rng.shuffle(cardsByLevel[level].map((c) => c.id));
    game.board.setLevelCards(level, shuffled.splice(0, 4), shuffled);
  }

  const nobleCount = playerIds.length + 1;
  game.board.setNobles(rng.shuffle(allNobles.map((n) => n.id)).slice(0, nobleCount));
}
```

`rng.shuffle()` is deterministic — the same RNG seed always produces the same shuffle, which is important for replay and testing.

</Step>

<Step title="Assemble the game definition">

Pull everything together with `GameDefinitionBuilder` and produce an executor with `createGameExecutor()`:

```typescript
import { createGameExecutor, GameDefinitionBuilder, t } from "tabletop-engine";
import { GameState } from "./state.ts";
import { setupGame } from "./setup.ts";
import { createStages } from "./stages/index.ts";

export function createGame() {
  const { initialStage } = createStages();

  return new GameDefinitionBuilder("my-game")
    .rootState(GameState)
    .setupInput(
      t.object({
        playerIds: t.array(t.string()),
      }),
    )
    .setup(({ game, rng, input }) => {
      if (input.playerIds.length < 2) {
        throw new Error("requires_at_least_two_players");
      }
      setupGame(game, rng, input.playerIds);
    })
    .initialStage(initialStage)
    .build();
}

export function createGameExecutorInstance() {
  return createGameExecutor(createGame());
}
```

The executor exposes all the methods you need to drive the game:

```typescript
const executor = createGameExecutorInstance();

// Create a fresh game state
const state = executor.createInitialState(
  { playerIds: ["alice", "bob"] },
  "my-rng-seed",
);

// List what the active player can do
const available = executor.listAvailableCommands(state, { actorId: "alice" });

// Execute a command and get the new state plus events
const result = executor.executeCommand(state, {
  type: "take_three_distinct_gems",
  actorId: "alice",
  input: { colors: ["white", "blue", "green"] },
});

if (result.ok) {
  console.log(result.state.game.playerOrder);
  console.log(result.events); // domain + runtime events emitted
} else {
  console.log(result.reason); // validation failure reason
}
```
