---
title: "Schema Types: the t Schema Builder Reference"
description: "Complete reference for the t schema builder: t.number, t.string, t.object, t.array, t.record, t.optional, and t.state with TypeScript inference details."
---

The `t` object is the runtime schema builder for `tabletop-engine`. You use it wherever the engine needs to know the shape of data: in `@field()` decorators on state classes, in `setupInput()` configuration, and in command schema definitions. Each method on `t` returns a typed field descriptor that the engine uses both for runtime validation and for TypeScript inference.

## Import

```typescript
import { t } from "tabletop-engine";
```

## Primitive types

### `t.number()`

```typescript
t.number(): NumberFieldType
```

Declares a floating-point number field. Inferred TypeScript type: `number`.

```typescript
@field(t.number())
score!: number;
```

### `t.string()`

```typescript
t.string(): StringFieldType
```

Declares a UTF-8 string field. Inferred TypeScript type: `string`.

```typescript
@field(t.string())
playerId!: string;
```

### `t.boolean()`

```typescript
t.boolean(): BooleanFieldType
```

Declares a boolean field. Inferred TypeScript type: `boolean`.

```typescript
@field(t.boolean())
isActive!: boolean;
```

## Composite types

### `t.object(properties)`

```typescript
t.object<TProperties extends Record<string, FieldType>>(
  properties: TProperties,
): ObjectFieldType<TProperties>
```

Declares an inline object field with named properties. The engine enforces that no extra properties are present at runtime.

<ParamField path="properties" type="Record<string, FieldType>" required>
  A map from property name to field type descriptor. Each value must itself
  be a `t.*` descriptor.
</ParamField>

```typescript
// t.object infers the shape from its properties
const schema = t.object({
  name: t.string(),
  score: t.number(),
  items: t.array(t.string()),
});
// Inferred static type: { name: string; score: number; items: string[] }

@field(t.object({ x: t.number(), y: t.number() }))
position!: { x: number; y: number };
```

### `t.array(item)`

```typescript
t.array<TItem extends FieldType>(item: TItem): ArrayFieldType<TItem>
```

Declares an ordered list of a single item type.

<ParamField path="item" type="FieldType" required>
  The field type for every element in the array.
</ParamField>

```typescript
@field(t.array(t.string()))
playerIds!: string[];

@field(t.array(t.number()))
cardIds: number[] = [];
```

### `t.record(key, value)`

```typescript
t.record<TKey extends PrimitiveFieldType, TValue extends FieldType>(
  key: TKey,
  value: TValue,
): RecordFieldType<TKey, TValue>
```

Declares a key-value map. The key must be a primitive type (`t.string()`, `t.number()`, or `t.boolean()`).

<ParamField path="key" type="PrimitiveFieldType" required>
  Must be `t.string()`, `t.number()`, or `t.boolean()`. Using a composite
  type here is a type error.
</ParamField>

<ParamField path="value" type="FieldType" required>
  The field type for every value in the record. May be any field type,
  including `t.state()`.
</ParamField>

```typescript
@field(t.record(t.string(), t.number()))
tokenCounts!: Record<string, number>;

// Nested state record — common for per-player state
@field(t.record(t.string(), t.state(() => PlayerState)))
players!: Record<string, PlayerState>;
```

### `t.optional(item)`

```typescript
t.optional<TItem extends FieldType>(item: TItem): OptionalFieldType<TItem>
```

Declares a field that may be `undefined`. Use this instead of the TypeScript `?` shorthand whenever you need the engine to track the field in its schema.

<ParamField path="item" type="FieldType" required>
  The field type when the value is present.
</ParamField>

```typescript
@field(t.optional(t.array(t.string())))
winnerIds?: string[];

@field(t.optional(t.state(() => EndGameState)))
endGame?: EndGameState;
```

### `t.state(factory)`

```typescript
t.state(target: () => StateClass): NestedStateFieldType
```

Declares a nested `@State()` class field. The engine hydrates the referenced class from canonical data before your logic runs.

<ParamField path="target" type="() => StateClass" required>
  A factory function returning the target state class. Passing a factory
  instead of the class directly avoids circular reference issues when two
  state classes reference each other.
</ParamField>

```typescript
@field(t.state(() => BoardState))
board!: BoardState;

// Combine with t.optional for nullable nested state
@field(t.optional(t.state(() => EndGameState)))
endGame?: EndGameState;
```

<Warning>
  `t.state()` fields cannot appear inside `t.object()` or in visibility
  `schema` options. Use a top-level `@field(t.state(...))` or nest state inside
  `t.record()` and `t.array()` instead.
</Warning>

## Type reference table

| Builder | Inferred TypeScript type | Can be used as a `t.record` key |
|---|---|:---:|
| `t.number()` | `number` | Yes |
| `t.string()` | `string` | Yes |
| `t.boolean()` | `boolean` | Yes |
| `t.object({ ... })` | `{ [key]: type }` | No |
| `t.array(item)` | `item[]` | No |
| `t.record(key, value)` | `Record<key, value>` | No |
| `t.optional(item)` | `item \| undefined` | No |
| `t.state(() => Class)` | Class instance | No |

## Composition example

The following example shows deeply nested composition across all composite types:

```typescript
import { State, field, t } from "tabletop-engine";

@State()
class HandState {
  @field(t.array(t.number()))
  cardIds: number[] = [];
}

@State()
class PlayerState {
  @field(t.string())
  id!: string;

  @field(t.number())
  score = 0;

  @field(t.state(() => HandState))
  hand!: HandState;

  @field(t.optional(t.object({ reason: t.string() })))
  eliminated?: { reason: string };
}

@State()
class GameState {
  @field(t.array(t.string()))
  playerOrder: string[] = [];

  @field(t.record(t.string(), t.state(() => PlayerState)))
  players: Record<string, PlayerState> = {};

  @field(t.record(t.string(), t.number()))
  bankTokens: Record<string, number> = {};

  @field(t.optional(t.array(t.string())))
  winnerIds?: string[];
}
```
