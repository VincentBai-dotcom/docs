---
title: "Command factory API reference"
description: "createCommandFactory returns a typed command builder. Chain .isAvailable(), .validate(), .execute(), and optionally .discoverable() before calling .build()."
---

Commands are the actions players send to the engine. `createCommandFactory` returns a function you call once per command to configure its ID, input schema, validation logic, and execution behavior. The builder enforces via TypeScript's type system that you call both `.validate()` and `.execute()` before `.build()` is available.

## `createCommandFactory<GameState>()`

```typescript
import { createCommandFactory } from "tabletop-engine";

const command = createCommandFactory<MyGameState>();
```

<ParamField path="GameState" type="generic type parameter" required>
  The facade game state type inferred from your root `@State()` class. All command callbacks receive the correctly typed game view.
</ParamField>

<ResponseField name="returns" type="CommandFactory<GameState>">
  A `defineCommand` function with the signature `(config: CommandBuilderBaseConfig) => CommandBuilder`. Call it once per command definition.
</ResponseField>

## Defining a command

Call the factory function with a base config to start a `CommandBuilder`:

```typescript
const placeToken = command({
  commandId: "place-token",
  commandSchema: t.object({
    position: t.number(),
  }),
})
  .validate(({ game, command }) => {
    if (game.board[command.input.position] !== null) {
      return { ok: false, reason: "position_occupied" };
    }
    return { ok: true };
  })
  .execute(({ game, command }) => {
    game.board[command.input.position] = command.actorId;
  })
  .build();
```

### Base config

<ParamField path="config.commandId" type="string" required>
  A unique string identifier for this command across the entire game. The engine uses this as the `type` field in `Command` objects and as the key in the game's command map. Duplicate IDs across stages cause `GameDefinitionBuilder.build()` to throw.
</ParamField>

<ParamField path="config.commandSchema" type="ObjectFieldType<Record<string, FieldType>>" required>
  A serializable object schema built with `t.object(...)` describing the command's input shape. The schema must be an object schema (kind `"object"`); the factory validates this at definition time. The schema's static type drives the `TCommandInput` type parameter used throughout the builder.
</ParamField>

## Methods on `CommandBuilder`

### `.isAvailable(fn)` <span style={{color:"grey"}}>optional</span>

Registers an availability check run by `listAvailableCommands`. If you omit this method, the command is always included in the available commands list when the stage is active.

<ParamField path="fn" type="(context: CommandAvailabilityContext) => boolean" required>
  Called for each candidate command when `listAvailableCommands` is invoked. Return `true` to include the command, `false` to hide it.

  The context provides:

  - `context.game` — a read-only facade view of the current game state
  - `context.runtime` — the current runtime state (read-only)
  - `context.commandType` — the `commandId` string for this command
  - `context.actorId` — the player ID whose available commands are being listed
</ParamField>

<ResponseField name="returns" type="CommandBuilder (re-typed)">
  The same builder with `HasAvailability` set to `true`. Once called, `.isAvailable()` is no longer present on the builder.
</ResponseField>

### `.validate(fn)` <span style={{color:"red"}}>required</span>

Registers validation logic run immediately before execution. Validation receives a read-only view of the game state and may reject the command with a custom reason.

<ParamField path="fn" type="(context: ValidationContext) => ValidationOutcome" required>
  The context provides:

  - `context.game` — a **read-only** facade view of the current game state
  - `context.runtime` — the current runtime state (read-only)
  - `context.command` — the typed command being validated, including `type`, `actorId`, and `input`

  Return `{ ok: true }` to allow execution, or `{ ok: false, reason: string, metadata?: unknown }` to reject it.
</ParamField>

<ResponseField name="returns" type="CommandBuilder (re-typed)">
  The builder with `HasValidate` set to `true`. Once called, `.validate()` is removed from the builder.
</ResponseField>

```typescript
.validate(({ game, command }) => {
  if (command.input.position < 0 || command.input.position >= game.board.length) {
    return { ok: false, reason: "position_out_of_bounds", metadata: { position: command.input.position } };
  }
  return { ok: true };
})
```

### `.execute(fn)` <span style={{color:"red"}}>required</span>

Registers the execution function that mutates the game state. Execute runs only after validation succeeds.

<ParamField path="fn" type="(context: ExecuteContext) => void" required>
  The context provides:

  - `context.game` — a **mutable** facade view of the game state
  - `context.runtime` — the current runtime state (read-only)
  - `context.command` — the typed command being executed, including `type`, `actorId`, and `input`
  - `context.rng` — the deterministic RNG service (see [RNG in execute context](#rng-in-execute-context))
  - `context.emitEvent(event)` — emits a `GameEvent` into the result's event list

  Mutate `context.game` directly. Do not return a value.
</ParamField>

<ResponseField name="returns" type="CommandBuilder (re-typed)">
  The builder with `HasExecute` set to `true`. Once called, `.execute()` is removed from the builder.
</ResponseField>

```typescript
.execute(({ game, command, rng }) => {
  game.board[command.input.position] = command.actorId;
  game.lastMove = { player: command.actorId, position: command.input.position };
})
```

### `.discoverable(configure)` <span style={{color:"grey"}}>optional</span>

Adds a multi-step discovery definition to the command. Discovery lets players navigate options across multiple round-trips before the final command input is assembled.

<ParamField path="configure" type="(step: DiscoveryStepFactory) => DiscoveryStepDefinition[]" required>
  A function that receives a `step` factory and returns an ordered tuple of step definitions. Each step is built with the step factory using a fluent sub-builder:

  ```typescript
  (step) => [
    step("select-card")
      .initial()
      .input(t.object({ handIndex: t.number() }))
      .output(t.object({ cardId: t.string() }))
      .resolve(({ game, input }) => {
        const card = game.hand[input.handIndex];
        if (!card) return null;
        return [{ id: card.id, output: { cardId: card.id }, nextStep: "select-target", nextInput: {} }];
      })
      .build(),
    step("select-target")
      .input(t.object({}))
      .output(t.object({ targetId: t.string() }))
      .resolve(({ game, input }) => ({
        complete: true,
        input: { cardId: ..., targetId: ... },
      }))
      .build(),
  ]
  ```

  Exactly one step must call `.initial()` to mark it as the entry point. Step IDs must be unique within the command.

  Each step sub-builder has:
  - `.initial()` — marks this step as the discovery entry point (required on exactly one step)
  - `.input(schema)` — sets the input schema for this step
  - `.output(schema)` — sets the output schema returned in the options list
  - `.resolve(fn)` — returns either an array of `DiscoveryStepOption` objects or `{ complete: true, input }` when discovery is finished
</ParamField>

<ResponseField name="returns" type="CommandBuilder (re-typed)">
  The builder with `HasDiscovery` set to `true`. Once called, `.discoverable()` is no longer present on the builder.
</ResponseField>

### `.build()` <span style={{color:"red"}}>required</span>

Produces the sealed `DefinedCommand`.

**Throws** `"command_builder_missing_validate"` or `"command_builder_missing_execute"` if the corresponding methods have not been called. TypeScript also prevents `.build()` from appearing on the builder type until both are satisfied.

<ResponseField name="returns" type="DefinedCommand<GameState, TCommandInput>">
  A branded command definition object. Pass this in the `.commands([...])` array of a stage builder.
</ResponseField>

## Context types reference

### `ValidationContext`

<ResponseField name="game" type="Readonly<FacadeGameState>">
  A read-only hydrated facade view of the current game state.
</ResponseField>

<ResponseField name="runtime" type="Readonly<RuntimeState>">
  The current runtime state, including `progression` (current stage, last acting stage), `rng` (seed and cursor), and `history` (executed command entries).
</ResponseField>

<ResponseField name="command" type="Command<TCommandInput>">
  The command being validated. Contains `type: string`, `actorId: string`, and `input: TCommandInput`.
</ResponseField>

### `ExecuteContext`

Extends `ValidationContext` with mutable game access and additional execution tools.

<ResponseField name="game" type="FacadeGameState">
  A mutable hydrated facade view of the game state. Changes you make here are persisted to the canonical state after execution completes.
</ResponseField>

<ResponseField name="runtime" type="Readonly<RuntimeState>">
  The current runtime state (read-only).
</ResponseField>

<ResponseField name="command" type="Command<TCommandInput>">
  The command being executed.
</ResponseField>

<ResponseField name="rng" type="RNGApi">
  The deterministic RNG service. See [RNG in execute context](#rng-in-execute-context).
</ResponseField>

<ResponseField name="emitEvent" type="(event: GameEvent) => void">
  Emits a custom event that appears in `ExecutionSuccess.events`.
</ResponseField>

### `CommandAvailabilityContext`

<ResponseField name="game" type="Readonly<FacadeGameState>">
  A read-only facade view of the current game state.
</ResponseField>

<ResponseField name="runtime" type="Readonly<RuntimeState>">
  The current runtime state (read-only).
</ResponseField>

<ResponseField name="commandType" type="string">
  The `commandId` of the command being checked.
</ResponseField>

<ResponseField name="actorId" type="string">
  The player ID whose available commands are being listed.
</ResponseField>

## `ValidationOutcome`

Return one of these shapes from your `.validate()` function:

```typescript
// Allow the command
return { ok: true };

// Reject with a reason
return {
  ok: false,
  reason: "position_occupied",
  metadata: { position: 5 },
};
```

<ResponseField name="ok" type="boolean" required>
  `true` to proceed to execution; `false` to reject with a failure result.
</ResponseField>

<ResponseField name="reason" type="string">
  Required when `ok` is `false`. A machine-readable string returned as `ExecutionFailure.reason`.
</ResponseField>

<ResponseField name="metadata" type="unknown">
  Optional extra context returned as `ExecutionFailure.metadata`. Use this to pass structured diagnostic data to the client.
</ResponseField>

## RNG in execute context

The `rng` object in `ExecuteContext` provides a deterministic, cursor-backed RNG. The cursor is persisted in `runtime.rng.cursor` so replaying the same sequence of commands always produces the same random values.

```typescript
.execute(({ game, rng }) => {
  // Shuffle an array in place, advancing the cursor for each element
  game.deck = rng.shuffle(game.deck);

  // Pick a random element from an array
  game.startingPlayer = rng.pick(game.playerIds);
})
```

Always prefer `context.rng` over `Math.random()` inside execute functions; using `Math.random()` breaks replay determinism.

## Full example

```typescript
import { createCommandFactory, t } from "tabletop-engine";
import { MyGameState } from "./state";

const command = createCommandFactory<MyGameState>();

export const placeToken = command({
  commandId: "place-token",
  commandSchema: t.object({
    position: t.number(),
  }),
})
  .isAvailable(({ game, actorId }) => game.tokens[actorId] > 0)
  .validate(({ game, command }) => {
    if (game.board[command.input.position] !== null) {
      return { ok: false, reason: "position_occupied" };
    }
    return { ok: true };
  })
  .execute(({ game, command }) => {
    game.board[command.input.position] = command.actorId;
    game.tokens[command.actorId] -= 1;
  })
  .build();
```
