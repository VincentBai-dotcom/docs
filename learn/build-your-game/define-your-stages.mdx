---
title: "Stages: Modeling Turn Progression as a Graph"
description: "Stages are nodes in a progression graph. Each stage declares who can act, what commands are available, and which stage comes next after a command executes."
---

A stage is a node in your game's progression graph. Every game session is always in exactly one stage, and each stage determines who is allowed to act, which commands they may submit, and where the engine transitions after the action resolves. You define stages using `createStageFactory()` and wire them together by passing stage references through `nextStages` resolvers.

## Creating a stage factory

Call `createStageFactory()` once per game, parameterised with your root state class. The factory returns a function you call with a stage ID to begin building a stage:

```typescript
import { createStageFactory } from "tabletop-engine";
import type { MyGameState } from "./state.ts";

const stage = createStageFactory<MyGameState>();
```

The `stage` function accepts a string ID and returns an object with three methods — one per stage kind.

## Stage kinds

### `singleActivePlayer`

Use this when exactly one player acts at a time. You provide a function that selects the active player, the commands they may submit, and a transition that runs after their command executes.

```typescript
const playerTurnStage = stage("player-turn")
  .singleActivePlayer()
  .activePlayer(({ game, runtime }) => game.currentPlayerId)
  .commands([drawCommand, playCardCommand])
  .nextStages(() => ({
    endTurn: endTurnStage,
    victory: victoryStage,
  }))
  .transition(({ game, command, nextStages }) => {
    if (game.hasWinner()) return nextStages.victory;
    return nextStages.endTurn;
  })
  .build();
```

**Required methods:**
- `.activePlayer(fn)` — receives `{ game, runtime }` and returns the player ID of the active player.
- `.commands([...])` — the commands available to the active player during this stage.
- `.nextStages(fn)` — a resolver that returns a named map of stages this stage can transition to.
- `.transition(fn)` — receives `{ game, runtime, command, nextStages }` and returns the next stage.

### `automatic`

Use this for engine-driven logic that runs without player input. An automatic stage can execute arbitrary game logic in `.run()` and then transition immediately. It does not require player commands.

```typescript
const checkVictoryStage = stage("check-victory")
  .automatic()
  .run(({ game, emitEvent }) => {
    game.evaluateVictoryConditions(emitEvent);
  })
  .nextStages(() => ({
    gameEnd: gameEndStage,
    playerTurn: playerTurnStage,
  }))
  .transition(({ game, nextStages }) => {
    return game.winnerIds ? nextStages.gameEnd : nextStages.playerTurn;
  })
  .build();
```

**Available methods:**
- `.run(fn)` — optional side-effectful logic that runs when the stage is entered. Receives `{ game, runtime, rng, emitEvent }`.
- `.nextStages(fn)` — a resolver returning the stages this stage can transition to.
- `.transition(fn)` — returns the next stage. Optional if the stage has no outgoing edges.

### `multiActivePlayer`

Use this when multiple players act simultaneously and the engine waits for all of them before moving on. Each player submits a command independently; the stage tracks shared memory across submissions.

```typescript
const bidStage = stage("simultaneous-bid")
  .multiActivePlayer()
  .memory(
    t.object({ bids: t.record(t.string(), t.number()) }),
    () => ({ bids: {} }),
  )
  .activePlayers(({ game }) => game.playerOrder)
  .commands([submitBidCommand])
  .onSubmit(({ game, memory, command, execute }) => {
    execute(command);
    memory.bids[command.actorId] = command.input.amount;
  })
  .isComplete(({ memory, game }) => {
    return game.playerOrder.every((id) => id in memory.bids);
  })
  .nextStages(() => ({ resolve: resolveStage }))
  .transition(({ nextStages }) => nextStages.resolve)
  .build();
```

**Required methods:**
- `.memory(schema, factory)` — declares the schema for the stage's shared memory object and a factory that produces its initial value. Memory is reset when the stage is entered.
- `.activePlayers(fn)` — returns the list of player IDs who must submit before the stage completes.
- `.commands([...])` — commands available to active players.
- `.onSubmit(fn)` — called each time a player submits a command. Call `execute(command)` inside this callback to actually apply the command.
- `.isComplete(fn)` — returns `true` when all required submissions are in. The engine polls this after each `onSubmit`.
- `.nextStages(fn)` — resolver returning the named map of stages to transition to.
- `.transition(fn)` — returns the next stage once `isComplete` is `true`.

## Stage transitions and the progression graph

The engine collects all reachable stages by walking the graph from `initialStage` at build time. Stage references in `nextStages` resolvers are the edges. Because `nextStages` is a function (not a plain object), you can close over lazily-evaluated references to handle cycles:

```typescript
// Stages that reference each other
const turnStage = stage("player-turn")
  .singleActivePlayer()
  // ...
  .nextStages(() => ({ next: checkVictoryStage }))   // closed over below
  .transition(({ nextStages }) => nextStages.next)
  .build();

const checkVictoryStage = stage("check-victory")
  .automatic()
  // ...
  .nextStages(() => ({ next: turnStage }))           // back-reference
  .transition(({ game, nextStages }) =>
    game.winnerIds ? gameEndStage : nextStages.next,
  )
  .build();
```

<Warning>
  All stage IDs in a game must be unique. `GameDefinitionBuilder.build()` throws a `duplicate_stage_id` error if two different stage objects share the same ID.
</Warning>

## Full example from Splendor

The Splendor implementation uses a factory function pattern to wire up the graph. Each stage is created inside a single factory so closures resolve correctly:

```typescript
import { createStageFactory } from "tabletop-engine";
import type { SplendorGameState } from "./state.ts";

export function createSplendorStages() {
  const defineStage = createStageFactory<SplendorGameState>();

  const gameEndStage = defineStage("game-end")
    .automatic()
    .build();

  const checkVictoryConditionStage = defineStage("check-victory-condition")
    .automatic()
    .run(({ game, runtime, emitEvent }) => {
      game.resolveTurnEnd(getLastActingPlayerId(runtime), emitEvent);
    })
    .nextStages(() => ({
      gameEndStage,
      playerTurnStage,
    }))
    .transition(({ game, nextStages }) =>
      game.winnerIds ? nextStages.gameEndStage : nextStages.playerTurnStage,
    )
    .build();

  const playerTurnStage = defineStage("player-turn")
    .singleActivePlayer()
    .activePlayer(({ game, runtime }) => {
      const prev = runtime.progression.lastActingStage;
      const prevId = prev?.kind === "activePlayer" ? prev.activePlayerId : null;
      return prevId ? game.getNextPlayerId(prevId) : game.playerOrder[0]!;
    })
    .commands(allPlayerCommands)
    .nextStages(() => ({ checkVictoryConditionStage }))
    .transition(({ nextStages }) => nextStages.checkVictoryConditionStage)
    .build();

  return { initialStage: playerTurnStage };
}
```
