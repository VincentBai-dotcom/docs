---
title: "State Facade: Authoring Game State with Decorators"
description: "Write game logic against typed class instances using @State() and @field() while the engine persists plain JSON — no manual serialization needed."
---

The state facade is a decorator-driven authoring layer that sits between your game logic and the engine's canonical state. You write business logic against typed TypeScript class instances — complete with methods, getters, and encapsulation — but the engine stores and transfers only plain JSON. The facade classes are hydrated from canonical data at execution time and discarded when execution ends.

## Defining a state class

Annotate any class with `@State()` and mark each data property with `@field(fieldType)`:

```typescript
import { State, field, t } from "tabletop-engine";

@State()
class PlayerState {
  @field(t.string())
  id!: string;

  @field(t.number())
  score!: number;

  @field(t.array(t.string()))
  hand!: string[];

  @field(t.boolean())
  isActive!: boolean;
}
```

`@State()` registers the class with the engine's metadata store. `@field(fieldType)` records the schema for each property so the engine knows how to serialize and deserialize it.

<Warning>
  Only properties annotated with `@field()` are included in the canonical schema. Any unannotated property is treated as transient and will not be persisted.
</Warning>

## The `t` schema builder

The `t` object provides factory functions for every supported field type:

| Builder | TypeScript type | Description |
|---|---|---|
| `t.string()` | `string` | UTF-8 string |
| `t.number()` | `number` | Floating-point number |
| `t.boolean()` | `boolean` | Boolean |
| `t.object({ ... })` | `{ ... }` | Inline object with named properties |
| `t.array(item)` | `item[]` | Ordered list of a single item type |
| `t.record(key, value)` | `Record<key, value>` | String-keyed map |
| `t.optional(item)` | `item \| undefined` | Nullable field |
| `t.state(factory)` | State class instance | Nested `@State()` class |

### Nesting state classes

Use `t.state()` to embed one `@State()` class inside another. Pass a factory function (not the class directly) to avoid circular reference issues:

```typescript
@State()
class GameState {
  @field(t.record(t.string(), t.state(() => PlayerState)))
  players: Record<string, PlayerState> = {};

  @field(t.state(() => BoardState))
  board!: BoardState;

  @field(t.optional(t.array(t.string())))
  winnerIds?: string[];
}
```

The `t.state()` factory tells the engine that the canonical representation of that field is itself a recursively structured object, and the engine will hydrate the correct class when you access it in command or stage logic.

## Canonical vs. facade state

The engine maintains two representations of state:

- **Canonical state** — plain JSON stored and transferred by the engine. All field values are primitives, arrays, or nested plain objects.
- **Facade state** — class instances the engine hydrates from canonical data before running your logic. This is what you read from and write to in `execute()`, `validate()`, `run()`, and other lifecycle functions.

When you mutate a facade instance inside `execute()`, the engine serialises those changes back to canonical form after your function returns. You never call a serialisation method yourself.

<Info>
  Methods on your state classes — getters, helpers, business logic — are available inside lifecycle functions but are never included in canonical state. Put all game logic that belongs in your classes there freely.
</Info>

## Complete example

Here is how the Splendor reference game defines its root state:

```typescript
import { field, State, t } from "tabletop-engine";
import { SplendorBoardState } from "./board-state.ts";
import { SplendorPlayerState } from "./player-state.ts";
import { TokenCountsState } from "./token-counts-state.ts";
import { SplendorEndGameState } from "./end-game-state.ts";

@State()
export class SplendorGameState {
  @field(t.array(t.string()))
  playerOrder: string[] = [];

  @field(t.record(t.string(), t.state(() => SplendorPlayerState)))
  players: Record<string, SplendorPlayerState> = {};

  @field(t.state(() => TokenCountsState))
  bank!: TokenCountsState;

  @field(t.state(() => SplendorBoardState))
  board!: SplendorBoardState;

  @field(t.optional(t.state(() => SplendorEndGameState)))
  endGame?: SplendorEndGameState;

  @field(t.optional(t.array(t.string())))
  winnerIds?: string[];

  // Methods are available in command and stage logic
  getPlayer(playerId: string): SplendorPlayerState {
    const player = this.players[playerId];
    if (!player) throw new Error(`unknown_player:${playerId}`);
    return player;
  }
}
```

## Default values

The engine derives a default canonical state from your class by instantiating it with `new` and reading the property initializers. Properties with `!` (definite assignment) must be initialised separately in a `setup()` callback. Properties with `= value` initializers produce the defaults automatically.

<Tip>
  Use `= []`, `= {}`, and `= ""` for zero-value initializers on array, record, and string fields. Reserve the `setup()` callback on `GameDefinitionBuilder` for values that depend on player count or other runtime input.
</Tip>
